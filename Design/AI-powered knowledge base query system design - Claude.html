Here's the complete design and architecture breakdown:

What you're looking at

Screen 1 is a full working mockup of the product — the sidebar shows your hierarchical folder structure (folders, subfolders, individual documents), the main panel handles querying, and answers come back with cited sources and a confidence score. The scope selector lets a user restrict queries to a subset of documents (e.g. "only search the Sign-In/Out folder").

Screen 2 is the system architecture — five layers from browser to storage, all clickable to learn more about each component.

Architecture: the five layers explained

Layer 1 — Clients. A web SaaS app is the primary interface. An admin portal handles document uploads and folder management. Mobile is optional but straightforward to add.

Layer 2 — Ingest & API. Everything enters through a single API Gateway with JWT/SSO authentication. Two ingest paths: a Document Pipeline for manual uploads (parses PDF/DOCX, cleans text, splits into chunks), and a Folder Sync Agent that watches a connected folder hierarchy (like a network drive or SharePoint) and automatically re-indexes changed files.

Layer 3 — AI Core. This is the RAG (Retrieval-Augmented Generation) pipeline. When a question arrives, it's converted to a vector embedding, the vector database is searched for the top 5 most semantically similar chunks, those chunks are fed to the LLM as context, and the LLM generates a grounded answer — meaning it can only answer from what's in the documents. A Re-ranker improves result quality, and the Citation Builder tracks exactly which document and page each part of the answer came from.

Layer 4 — Storage. Three stores working together: a Vector Database (Pinecone or Weaviate) holding the semantic index, a Document Store (S3/Azure Blob) holding original files and extracted text, and a relational Metadata DB (PostgreSQL) holding folder structure, tags, version history, and access permissions.

Layer 5 — Ops. An Analytics Engine tracks which questions get asked, which get poor confidence scores (i.e. gaps in your knowledge base), and usage patterns. A Semantic Cache in Redis means identical or near-identical questions don't hit the LLM again, saving cost significantly. The Feedback Loop captures thumbs up/down on answers and feeds that back as a retraining signal over time.

Key design decisions for your case

No internet leakage. The LLM is given a strict system prompt: "only answer from the provided context. If the answer isn't in the documents, say so." This is enforced architecturally — the model receives only document chunks, never a search tool.

Hierarchical folder structure preserved. The metadata database stores the full folder tree, so users can scope queries to a specific folder/subfolder, or to a tagged collection. This replaces Helpjuice's rigid categories with flexible, scoped search.

Replaces Helpjuice's limitation. Helpjuice requires structured articles. This system lets a user ask any free-form question and synthesises the answer across multiple documents — something Helpjuice structurally cannot do.

Cost control. The semantic cache means repeat questions (e.g. "what is your pricing?") are served from cache without an LLM call. For a support use case, 40–60% of questions are typically repetitive, making this highly effective at reducing API costs.

For the tech stack I'd recommend: Next.js frontend, Node/Python API layer, Claude Sonnet 4 or GPT-4o as the LLM, Pinecone for vectors, PostgreSQL + S3 on AWS or Azure.

The vector database and the document store do two different jobs. The vector DB is like a highly sophisticated index — it tells you which chunks are relevant (returning just IDs and similarity scores). It doesn't store your actual document text. The LLM only enters the picture at step 6, and by then it's working with a very tight, pre-filtered set of 3–5 passages — which is why answers are fast, grounded, and citable rather than invented.

Good question. Let me give you an honest critique of the v4 diagram.

What the diagram does well

The overall flow is architecturally sound. The two-pipeline split (ingest vs query), the RAG zone, the cloud boundary, the observability sidebar, and the toggle filters all make it genuinely readable. The ID linking annotations (chunk_id → doc_id + version_id) directly addressed a real gap from earlier versions.

Honest criticisms

1. The upsert return arrow is misleading

The diagram shows a dashed arrow going from the doc store back to the ingestion orchestrator labelled "upsert / version bump". This is backwards. The orchestrator instructs the doc store to upsert — it doesn't receive a signal back from the doc store. The arrow direction implies the doc store is triggering the orchestrator, which is false. It should either be removed or redrawn as a bidirectional annotation, not a return arrow.

2. Cache + rate limit are two entirely different concerns shown as one box

Caching (semantic similarity cache, Redis) and rate limiting (API quota enforcement, token bucket) are operationally distinct systems. Grouping them in one green box implies they live in the same service — they don't in any real architecture. Rate limiting typically sits at the API gateway layer, before the query even reaches the retriever. Caching sits inside the retrieval path. Showing them together in the retriever zone is architecturally inaccurate.

3. Auth/session check and access control are also conflated at the wrong layer

Authentication (who are you — JWT validation) should happen at the API gateway or SaaS endpoint layer, not inside the RAG pipeline. The diagram places it inside or adjacent to the retriever service, which suggests the RAG pipeline is doing auth. In a real system, by the time a request reaches the retriever, auth is already done. The request arrives with a validated user context, not a raw JWT.

4. The embedding model is shown as a local server-side component during ingestion but the cloud boundary is never crossed

During ingestion, the chunker sends text to the embedding model, which converts it to vectors. But if you're using Anthropic or OpenAI embeddings (as the diagram implies with "1536-dim vector"), that embedding call also crosses the cloud boundary — just like the LLM call does. The diagram never shows this. The cloud boundary box only contains the LLM and storage layer, making it appear embeddings are computed locally. This is false for any hosted embedding model.

5. The retriever service zone contains the prompt builder and re-ranker, which don't belong there

The retriever service's job is to find relevant chunks. The re-ranker and prompt builder are post-retrieval steps — they transform and package what the retriever found. Putting them inside the retriever zone blurs the conceptual boundary between retrieval and generation, which is the most important architectural distinction in RAG.

6. Metadata DB read is missing from the query flow

During ingestion, metadata is written (doc name, folder, page, version). But during the query flow, that metadata needs to be read back to populate the citation panel (which document, which page). The diagram shows the doc store being read but the metadata DB read is absent from the query path. The citation formatter cannot produce "Sign-In Guide p.12" without reading the metadata DB.

7. The observability sidebar is disconnected

The dashed line from observability connects only to the citation formatter. In reality, observability should tap into multiple points — the LLM response (confidence score), the re-ranker (retrieval quality), the cache (hit/miss ratio), and the job queue (ingestion failures). A single connection to the formatter makes it look like observability only watches the final output.

8. No error / fallback path shown

What happens when the retriever finds nothing above a confidence threshold? What happens when the LLM call fails or times out? Real systems have fallback responses ("I couldn't find an answer in the knowledge base") but the diagram shows only the happy path. For a production SaaS this is a meaningful omission.

9. "Hybrid search" is labelled but never explained structurally

The node says "Hybrid search + meta filter — vector + keyword". But keyword search requires a separate index (BM25/Elasticsearch). The diagram doesn't show this index existing anywhere — not in the storage layer, not during ingestion. If you claim hybrid search, you need a keyword index write during ingestion and a keyword index read during query, neither of which appear.

Summary verdict

The diagram is a good conceptual overview but would mislead a developer trying to build from it. The three most critical fixes are: move auth to the API gateway, show the embedding API call crossing the cloud boundary, and add the metadata DB read in the query flow. The rest are refinements.
